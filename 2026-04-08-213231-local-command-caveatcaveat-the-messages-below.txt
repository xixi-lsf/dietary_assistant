  ---
  当前 Agent 特性在哪里

  你的 app 已经有一些 agent 的影子：

  1. 上下文感知决策 —
  推荐菜谱时自动拉取冰箱库存、营养日志、用户偏好、历史反馈，而不是让用户手动填表
  2. 记忆写回循环 — extract_feedback_tags() 把用户反馈转化为偏好标签写回
  profile，下次推荐时生效，这是一个简单的 memory loop
  3. 多步骤任务链 — generate_recipe_detail()
  先生成步骤文本，再为每步提交图片生成任务，轮询结果，这是一个 mini workflow
  4. 个性化系统提示 — chat 端点动态构建 system
  prompt（含今日热量、冰箱内容），每次对话都是有状态的

  ---
  和传统 APP 的本质区别

  传统 APP：用户输入 → 固定逻辑 → 固定输出
  你的 APP：用户输入 → 模型推理 + 动态上下文 → 因人而异的输出

  关键差异是输出不可预测且个性化，这是规则引擎做不到的。

  ---
  现在缺什么让它"更 Agent"

  目前它本质上还是 prompt orchestration，不是真正的 agent。差距在：

  - 没有 tool calling — 模型不能主动决定"我需要查一下冰箱"，是代码硬编码去查的
  - 没有 planning loop — 模型不能把"帮我准备一周饮食计划"拆解成子任务自主执行
  - 没有 跨会话持久记忆 — 对话历史存在 SharedPreferences，不是真正的长期记忆

  ---
  怎么让它有研究性

  这取决于你想研究什么方向，几个有价值的角度：

  方向 A：个性化 + 记忆
  研究问题：长期记忆如何影响饮食建议质量？
  做法：设计结构化用户模型（不只是 tag list），对比有/无记忆的推荐满意度，量化
  feedback loop 的收敛速度

  方向 B：Multi-agent 协作
  研究问题：专业化 agent 分工是否优于单一通用 agent？
  做法：拆成营养师 agent、厨师 agent、采购 agent，用 orchestrator 协调，对比单
  agent 基线

  方向 C：Human-in-the-loop
  研究问题：用户反馈介入频率如何影响系统适应速度？
  做法：设计不同反馈粒度（点赞/评分/文字），测量偏好模型收敛

  方向 D：Tool use 的实际效果
  研究问题：给模型真正的 tool calling 能力后，推荐链路质量如何变化？
  做法：把冰箱查询、营养计算、菜谱搜索做成真正的

  tools = [
      {"name": "get_fridge_contents", ...},

> 是否需要区分长期记忆和短期记忆？长期记忆包含用户画像、历史反馈，短期记忆包含  
用户饮食观察周期内的饮食情况、历史对话                                          

● 这个区分非常有价值，而且在你的系统里天然就存在这两种数据，只是没有显式建模。

  ---
  长短期记忆的区别

  ┌──────────┬─────────────────────────────────┬───────────────────────────┐
  │          │            长期记忆             │         短期记忆          │
  ├──────────┼─────────────────────────────────┼───────────────────────────┤
  │ 内容     │ 用户画像、口味偏好权重、硬约束  │ 当前周期饮食情况、当前会  │
  │          │ 、历史反馈摘要                  │ 话对话历史                │
  ├──────────┼─────────────────────────────────┼───────────────────────────┤
  │ 更新频率 │ 慢，每次反馈后小幅更新          │ 快，每餐/每条消息更新     │
  ├──────────┼─────────────────────────────────┼───────────────────────────┤
  │ 生命周期 │ 永久，带衰减                    │ 一个观察周期（cycle_days  │
  │          │                                 │ ）或一次会话              │
  ├──────────┼─────────────────────────────────┼───────────────────────────┤
  │ 作用     │ 塑造推荐风格和硬约束            │ 影响当下决策（今天吃太油  │
  │          │                                 │ 了就推清淡）              │
  ├──────────┼─────────────────────────────────┼───────────────────────────┤
  │ 对应现有 │ UserProfile.preferences +       │ NutritionLog + chat       │
  │ 数据     │ Feedback                        │ history                   │
  └──────────┴─────────────────────────────────┴───────────────────────────┘

  ---
  两者如何协作

  用户请求"今晚吃什么"
          ↓
  长期记忆：你不吃香菜，偏好清淡(0.8)，目标减脂
          +
  短期记忆：今天已摄入 1800kcal，午饭吃了红烧肉（高脂）
          ↓
  模型决策：推荐低脂、无香菜、控热量的晚餐

  短期记忆是"今天的状态"，长期记忆是"这个人是谁"。两者缺一不可。

  ---
  对研究的意义

  这个区分本身就是一个研究贡献点：

  ▎ RQ：长短期记忆的组合方式如何影响饮食推荐的个性化质量？

  可以设计消融实验：
  - 只用长期记忆
  - 只用短期记忆
  - 两者结合
  - 两者结合 + 显式冲突解决（长期偏好 vs 短期营养需求冲突时怎么办）

  最后一个最有意思——比如用户长期偏好重口味，但今天已经超标了，模型应该怎么平衡？
  这个偏好 vs 健康目标的张力是很好的研究问题。

  ---
  建议的数据结构

  长期记忆（持久化到 DB）
  {
    "taste_weights": {"spicy": 0.8, "light": 0.6, "sweet": 0.2},
    "hard_constraints": ["香菜", "内脏"],
    "health_goals": ["减脂", "控糖"],
    "feedback_summary": "偏好家常菜，不喜欢复杂步骤",
    "last_updated": "2026-04-08"
  }

  短期记忆（周期内动态计算，不单独存储）
  {
    "cycle_nutrition": {"avg_calories": 1950, "avg_fat": "偏高", "avg_protein":
  "不足"},
    "today_meals": [{"meal": "红烧肉", "calories": 650, "fat": "高"}],
    "recent_recipes": ["红烧肉", "炒青菜"],  // 避免重复推荐
    "session_context": [...]  // 当前对话历史
  }

  短期记忆不需要新建表，从现有的 NutritionLog 实时计算就行，只是需要在 prompt
  构建时显式组装进去。

  ---
  怎么查看收敛曲线

  启动应用后，底部导航栏点「记忆」tab（大脑图标，在管家和设置之间）。

  页面包含四个部分：

  1. 收敛指数卡片
  - 0~100% 的进度条，绿色=已收敛，橙色=学习中，灰色=数据不足
  - 显示累计反馈条数和文字摘要

  2. 口味权重收敛曲线
  - X轴：反馈次数，Y轴：权重值（0~1）
  - 每个口味标签一条折线，从 0.5 出发逐渐稳定
  - 触摸折线可看具体数值

  3. 权重方差（稳定性指标）
  - 方差下降 = 系统对你的偏好越来越确定
  - 这张图直接对应论文里的"收敛速度"分析

  4. 推荐满意度趋势
  - 散点=单次评分，折线=滑动平均（窗口3）
  - 折线上升 = 记忆系统在改善推荐质量

  ---
  怎么产生数据

  1. 去「菜单」tab 推荐几道菜
  2. 点菜卡上的「评价」→ 👍/👎 → 选原因
  3. 去「记忆」tab 刷新，曲线就会更新

  至少需要 3 条反馈才会出现折线，5 条以上收敛指数才有意义。

  ▎ RQ：让模型自主规划信息获取（tool calling）vs 硬编码

  4. 最后生成推荐
